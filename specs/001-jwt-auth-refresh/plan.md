# Implementation Plan: JWT Authentication & Authorization with Refresh Tokens

**Branch**: `001-jwt-auth-refresh` | **Date**: 2026-05-02 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-jwt-auth-refresh/spec.md`
**Implementation Model**: `claude-haiku-4-5-20251001` (user-specified)

## Summary

Add a complete authentication and authorization system to the Skyress e-commerce API. Users register and log in with email/password credentials and receive a short-lived JWT access token plus a long-lived opaque refresh token stored server-side. All existing business endpoints are protected. Role-based authorization distinguishes Admin from Customer. Refresh tokens rotate on each use and include family-based replay-attack detection.

## Technical Context

**Language/Version**: C# 12 / .NET 8.0
**Primary Dependencies**: ASP.NET Core 8, Entity Framework Core 8 (Npgsql/PostgreSQL), MediatR 12, FluentValidation 11, Microsoft.AspNetCore.Authentication.JwtBearer 8.0.0, BCrypt.Net-Next 4.0.3
**Storage**: PostgreSQL — refresh tokens and user records persisted in `SkyressDbContext`
**Testing**: No test projects exist — out of scope for this feature (add in a follow-up)
**Target Platform**: Linux server (Docker, existing docker-compose setup)
**Project Type**: Web API service — Clean Architecture (Domain / Application / Infrastructure / API)
**Performance Goals**: Token issuance ≤ 200 ms; token refresh ≤ 200 ms; login ≤ 500 ms
**Constraints**: Access tokens are stateless (JWT, not stored); refresh tokens are opaque and stored server-side; `Treat Warnings As Errors` is enabled in all projects
**Scale/Scope**: Single-tenant; supports multiple concurrent sessions per user
